# CloudBase 安全规则 v1.0.0

## 原则

小程序端不直接操作云数据库。读取和写入均通过云函数完成；云函数使用 `getWXContext()` 获取身份，并在服务端执行归属、状态、拉黑和角色判断。

## 推荐数据库权限

对所有业务集合设置：

```json
{
  "read": false,
  "write": false
}
```

适用集合：`users`、`pets`、`topics`、`posts`、`comments`、`likes`、`favorites`、`follows`、`notifications`、`moderation_jobs`、`reports`、`blocks`、`audit_logs`。

## 已实施的服务端权限

- 用户只能修改自己的资料、删除自己的动态和评论。
- 宠物 `ownerId`、动态 `authorId`、评论 `authorId` 由云函数写入。
- 动态只能关联当前用户自己的宠物。
- 待审核或拒绝动态仅作者和管理员可处理，公共接口不返回。
- 点赞、收藏和评论只作用于已审核公开动态。
- 禁言用户不能发布动态或评论；停用和注销用户不能调用主要业务云函数。
- 拉黑关系会阻止关注和部分互动，并过滤信息流与评论。
- 通知只能由接收用户读取和标记已读。
- 举报目标、所有者和摘要由服务端解析，客户端不能伪造。
- 管理员操作要求当前用户 `role=admin`、`status=active`，并写入 `audit_logs`。
- 人工审核结论优先于迟到的异步通过回调。

## 生产环境建议加强

- 对发布、评论、举报和关注增加用户级速率限制。
- 对点赞、收藏、关注使用唯一索引与事务，定期校准计数。
- 对账号注销、批量审核和用户状态变更增加二次确认。
- 控制 `callbackPayload` 的访问和保留期限，避免长期保存不必要数据。
- 增加云函数错误、内容安全失败、举报积压和审核超时告警。
- 管理员数量保持最小，并定期审计角色。

## 不应放在前端

- AppSecret、管理员密钥、数据库凭证
- 内容安全绕过开关
- 管理员角色判断的唯一依据
- 审核回调的信任逻辑
